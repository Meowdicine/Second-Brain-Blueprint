# Getting Started

## 1. Start Small

Do not redesign the whole system on day one. Keep the structure, replace the examples, and learn where your real friction appears.

## 2. Fork, Rename, Adapt

Suggested steps:

1. Fork or clone the repository.
2. Decide whether you want to keep the PARA-like folder names.
3. Replace example notes with your own low-risk notes first.
4. Rename only the folders or templates that genuinely mismatch your workflow.

## 3. Keep It Markdown-First

This repo assumes that markdown is the long-lived source of truth.

That means:

- write notes in plain `.md` files
- keep links human-readable
- avoid storing core knowledge in opaque app databases
- use Git commits for meaningful changes over time

## 4. Use AI Safely

AI works best here when it is helping you:

- scaffold structure
- clean messy notes
- extract reusable methods
- summarize projects at closure
- prepare reviews and planning docs

AI should not silently decide:

- what is safe to publish
- what should be deleted
- what private context can be exposed
- how your life categories should be restructured without review

## 5. Good First Customizations

- edit the templates in `99_System/Templates/`
- adapt the prompts in `99_System/Prompts/`
- rewrite the architecture note in `99_System/Maps/PERSONAL-OS.md`
- create your first project note and first weekly review

## 6. Keep The Workflow Lean

If a new automation, dashboard, or script does not clearly save time, skip it for now. The point of the system is continuity, not complexity.
